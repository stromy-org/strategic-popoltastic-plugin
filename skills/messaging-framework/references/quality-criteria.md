## Validation Checklist

Run through this checklist before presenting the final framework to the user. Flag any item that fails.

### Core Message

- [ ] Under 25 words
- [ ] Immediately understandable without industry jargon
- [ ] Cannot be credibly claimed by a direct competitor
- [ ] Addresses what the primary audience cares about (not what the company is proud of)
- [ ] Passes the "cocktail party test" — someone could repeat it after hearing it once
- [ ] States a claim, not a description (there's something at stake)

### Messaging Pillars

- [ ] 3-4 pillars (2 is too thin, 5+ is too many to remember)
- [ ] Each pillar covers distinct territory (no significant overlap)
- [ ] Together they build a complete case for the core message
- [ ] Each pillar has a clear headline (3-7 words) that's memorable on its own
- [ ] Each pillar has 2+ proof points from 2+ proof types
- [ ] No pillar relies entirely on Tier 5 evidence (unsubstantiated claims)
- [ ] Pillars are framed as benefits/outcomes, not features/capabilities

### Proof Points

- [ ] Every pillar has at least 2 proof points
- [ ] At least 3 different proof types used across the framework
- [ ] Strongest proof (Tier 1-2) is attached to the most important pillars
- [ ] No proof point is unverifiable or obviously self-serving
- [ ] Proof is recent (within 2 years for metrics, 3 years for case studies)
- [ ] Proof gap analysis completed and shared with user

### Audience Adaptations

- [ ] Primary audience clearly identified
- [ ] Each audience has a distinct profile (not just a different label on the same content)
- [ ] Adaptations change emphasis and framing, not the core message
- [ ] Each audience's top objection is addressed
- [ ] Vocabulary matches each audience's register (technical, business, consumer)
- [ ] No audience adaptation contradicts another (if so, the core message needs work)

### Verbal Guardrails

- [ ] Voice attributes are specific and distinctive (not "professional, innovative")
- [ ] "Language to avoid" list includes company-specific hazards, not just generic buzzwords
- [ ] Elevator pitch is 30-50 words and sounds natural when spoken aloud
- [ ] If tagline candidates are included, each is under 10 words

### Structural

- [ ] Framework type matches the user's stated need
- [ ] Document is formatted for usability (not a wall of text)
- [ ] "When to revisit" section included with specific triggers
- [ ] Content library updated (or update offered) if company data directory exists

## Common Pitfalls

The most frequent messaging framework failures, drawn from practitioner consensus. Use these as a negative checklist — if the framework exhibits any of these patterns, revise before presenting.

### 1. Self-Focused Messaging

**What it looks like:** Core message and pillars lead with "we" and describe the company's capabilities rather than the audience's outcomes.

**Test:** Read the core message. Replace the company name with a competitor's. If it still works, the messaging is generic.

**Fix:** Rewrite from the audience's perspective. Instead of "We deliver industry-leading digital transformation," try "Enterprises modernize their operations 3x faster." The subject of the sentence should be the audience, not the company.

### 2. Feature-First Language

**What it looks like:** Pillars describe what the product does (features) rather than what the audience achieves (outcomes).

**Test:** For each pillar, ask "So what?" If the answer is more interesting than the pillar itself, the pillar is a feature, not a benefit.

**Fix:** Rewrite each pillar as an outcome. "Automated task assignment" → "Teams reclaim 5 hours per week." The feature becomes a proof point under the benefit.

### 3. Vagueness and Buzzwords

**What it looks like:** Core message uses terms like "industry leader," "cutting-edge," "innovative solutions," "transformation," "best-in-class," "unlock potential," "empower," or "leverage" without specificity.

**Test:** Count vague adjectives. More than one per pillar is a red flag. If you can't explain what a phrase means to a teenager, it's too vague.

**Fix:** Replace every vague claim with a specific one. "Industry-leading platform" → "Used by 12 of the top 20 European banks." If you can't make it specific, the claim may not be true.

### 4. Trying to Speak to Everyone

**What it looks like:** The messaging has no clear primary audience. Pillars try to simultaneously address customers, investors, employees, and partners.

**Test:** Ask "who would be most disappointed if we removed this pillar?" If the answer is "nobody in particular," the pillar is trying to please everyone.

**Fix:** Pick one primary audience. Write for them first. Then adapt for secondary audiences. The best messaging speaks deeply to a few rather than shallowly to many.

### 5. Unsubstantiated Claims

**What it looks like:** Pillars make strong claims ("we reduce costs by 40%") with no proof points, or proof points are all Tier 5 (self-reported, unverifiable).

**Test:** For each claim, ask "says who?" If the only answer is "we do," the claim needs evidence.

**Fix:** Either find proof or soften the claim. "We reduce costs by 40%" with no evidence → either find case studies that demonstrate it, or reframe as "our approach is designed to reduce operational costs."

### 6. Static Mindset

**What it looks like:** Framework is produced as a one-time document with no guidance on when or how to update it.

**Test:** Does the framework include review triggers? Does it mention any conditions under which the messaging should change?

**Fix:** Include a "When to revisit" section with specific triggers (new competitors, product changes, market shifts, quarterly review cadence).

### 7. Overlong Documents

**What it looks like:** The framework is 20+ pages, includes every possible message variant, and no one on the team has read it.

**Test:** Can a new team member find the core message, pillars, and their audience's adaptation within 2 minutes?

**Fix:** Keep the primary framework document to 2-4 pages. Move detailed audience adaptations, proof libraries, and channel-specific variants to appendices or separate documents.

### 8. Pillar Overlap

**What it looks like:** Two pillars make essentially the same claim from slightly different angles. ("Speed" and "Efficiency" are often the same thing in disguise.)

**Test:** Can you merge two pillars without losing important information? If yes, they overlap.

**Fix:** Merge the overlapping pillars and add a new one that covers genuinely different territory.

### 9. Playing It Safe

**What it looks like:** Messaging is accurate but forgettable. It describes the category rather than differentiating within it. No one would disagree with it — but no one would remember it either.

**Test:** If the messaging appeared on a competitor's website, would their customers notice something was off? If not, it's too generic.

**Fix:** Push for a point of view. Good messaging implies its opposite — "we believe in depth over breadth" implies competitors are shallow. If nothing is excluded, nothing is included.

## Differentiation Tests

Run these tests to verify the core message and pillars are genuinely differentiated.

### The Competitor Swap Test

Replace your company name with a competitor's name in the core message. If the message still reads as plausible, the messaging isn't differentiated enough.

### The "So What" Chain

For the core message, ask "so what?" up to 3 times. Each answer should get more specific and more defensible. If you hit a dead end or a generic claim within 2 iterations, the message lacks depth.

### The Opposite Test

State the opposite of each pillar. If the opposite is absurd (nobody would ever claim the opposite), the pillar is too generic. Good pillars have defensible opposites.

- **Too generic:** "We deliver quality" (opposite: "we deliver poor quality" — no one says this)
- **Differentiated:** "We prioritize depth over speed" (opposite: "we prioritize speed over depth" — a legitimate competitor position)

### The Deletion Test

Remove one pillar at a time. If the framework still feels complete without it, that pillar isn't pulling its weight. Every pillar should leave a visible gap when removed.

## Framework Usability Check

A framework is only valuable if teams actually use it. Before finalizing, check usability.

### Can a team member pull language directly?

The framework should contain "copy-ready" language that someone can drop into an email, slide, or social post without rewriting.

### Can a new hire understand it in 5 minutes?

Hand the framework to someone unfamiliar with the company. If they can explain the core message and top 3 pillars after reading for 5 minutes, the framework is usable.

### Is it findable?

If saved to the messaging content library, is it discoverable by downstream skills and team members? Clear file names, consistent structure, and proper labeling matter.

### Is it maintainable?

Can someone update proof points, add an audience, or adjust a pillar without restructuring the entire document? The framework should be modular enough to evolve.

## Practitioner Quality Markers

Synthesis of quality criteria from practitioner sources, for reference.

| Source | Key Criterion |
|--------|--------------|
| **Wynter** | Hierarchy of effectiveness: clarity → relevance → value → differentiation |
| **Cascade Insights** | Differentiated messaging must be specific, valuable to the buyer, and backed by research |
| **McKinsey** | Ruthless prioritization — focus on few real value drivers, don't drown in detail |
| **UserEvidence** | Buyers rank statistical evidence and case studies as most trustworthy; want proof from similar customers |
| **Masthead Media** | If no one on the team references it, the framework failed regardless of how good the content is |
| **Weber Shandwick** | Stress-test messaging across stakeholder reactions before publication |
